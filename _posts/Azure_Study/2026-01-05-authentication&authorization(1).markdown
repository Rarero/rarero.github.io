웹 개발이나 보안을 공부하다 보면 가장 많이 접하면서도 헷갈리는 두 가지 개념이 있습니다. 바로 **인증**(Authentication)과 **인가**(Authorization)입니다.

이 두 용어는 비슷해 보이지만, 실제로는 전혀 다른 역할을 수행합니다. 이번 포스트에서는 이 두 개념의 정의와 차이점을 예시를 통해 확실하게 정리해보겠습니다.
<br>

## 1. 인증 (Authentication)

**"당신은 누구입니까?"**

인증은 **사용자가 주장하는 신원이 실제 본인임을 확인하는 절차**입니다. 쉽게 말해 로그인 과정이 가장 대표적인 인증입니다.

### 인증의 예시
- **로그인**: 아이디와 비밀번호를 입력하여 내가 가입한 사용자임을 증명합니다.
- **OTP (One Time Password)**: 은행 이체 시, 등록된 사용자가 맞는지 확인하기 위해 일회용 비밀번호를 입력합니다.
- **생체 인식**: 지문이나 Face ID로 본인임을 증명합니다.
- **신분증 확인**: 공항 검색대에서 여권을 보여주며 내가 탑승객임을 증명합니다.

### 웹에서의 인증 방법
- Session/Cookie 방식
- JWT (JSON Web Token)
- OAuth (구글, 카카오 로그인 등)

## 2. 인가 (Authorization)

**"당신은 무엇을 할 수 있습니까?"**

인가는 **인증된 사용자가 특정 리소스에 접근하거나 특정 작업을 수행할 수 있는 권한이 있는지를 확인하는 절차**입니다. 반드시 **인증** 과정이 선행되어야 합니다.

### 인가의 예시
- **관리자 페이지 접근**: 일반 사용자는 상품 목록만 볼 수 있지만, 관리자는 상품을 등록/수정/삭제할 수 있는 권한이 있습니다.
- **파일 권한**: 리눅스에서 특정 파일에 대해 `Read`, `Write` 권한이 있는지 확인합니다.
- **유료 회원 전용 콘텐츠**: 넷플릭스 등에서 구독(결제)을 한 사용자만 영상을 볼 수 있게 합니다.

### 웹에서의 인가 처리
- 사용자의 역할(Role)에 따른 접근 제어 (RBAC: Role-Based Access Control)
- 특정 리소스에 대한 소유권 확인 (예: 내 글만 수정 가능)

## 3. 인증 vs 인가 비교

가장 쉬운 비유는 "**건물 출입**"입니다.

| 구분 | &nbsp; 인증 (Authentication) | &nbsp; 인가 (Authorization) |
| :--- | :--- | :--- |
| **핵심 질문** | &nbsp; 당신은 누구인가? (Who are you?) | &nbsp; 당신은 무엇을 할 수 있는가? (What can you do?) |
| **목적** | &nbsp; 신원 확인 | &nbsp; 권한 부여 및 접근 제어 |
| **순서** | &nbsp; 먼저 수행됨 | &nbsp; 인증 이후에 수행됨 |
| **웹 상태 코드** | &nbsp; 401 Unauthorize (Unauthenticated가 더 정확한 표현) | &nbsp; 403 Forbidden |
| **비유 (회사)** | &nbsp; 사원증을 찍고 게이트 통과 (출근) | &nbsp; 사원증으로 임원실이나 서버실 출입 시도 (권한 확인) |

> **참고**: HTTP 상태 코드 `401 Unauthorized`는 이름 때문에 인가(Authorization) 문제인 것처럼 보일 수 있지만, 실제로는 **인증이 되지 않았음**을 의미합니다. 반면, 인증은 되었지만 권한이 없을 때는 `403 Forbidden`을 사용합니다.

## 4. 요약

1.  **인증**(Authentication)은 사용자의 신원을 입증하는 것 (로그인).
2.  **인가**(Authorization)는 사용자에게 권한을 부여하거나 거부하는 것 (접근 제어).
3.  시스템은 먼저 **인증**을 통해 사용자를 식별하고, 그 사용자가 요청한 리소스에 접근할 수 있는지 **인가** 과정을 통해 판단합니다.

이 두 개념을 명확히 구분하면 보안 시스템의 구조를 이해하고 설계하는 데 큰 도움이 됩니다.
